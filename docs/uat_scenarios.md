# Design Support System - 사용자 테스트 시나리오 (UAT)
**작성일시:** 2026-05-07
**버전:** 2.0.0

---

## 1. 테스트 개요

### 1.1. 테스트 목표
- 근거 기반 디자인 창작 파이프라인의 전 단계 검증
- 챗봇 협업 모드와 자동 진행 모드 동작 확인
- 사용자 스케치 업로드, AI 해석, 구체화 기능 검증
- 레퍼런스 검색, 추상화, 이미지 생성 파이프라인 검증
- 스펙 문서 생성 및 버전 관리 검증
- SaaS 멀티테넌시 격리 검증

### 1.2. 테스트 환경
- **브라우저**: Chrome, Firefox, Safari, Edge (최신 버전)
- **디바이스**: 데스크톱, 태블릿
- **도메인**: 산업디자인, 패션디자인, 시각디자인, 광고디자인

## 2. 기능 테스트 시나리오

### 2.1. 프로젝트/세션 관리

#### TC001: 프로젝트 생성과 세션 시작
**테스트 목적**: 프로젝트-세션 생성 및 도메인 선택 검증

**수행 단계**:
1. 워크스페이스 접속
2. 새 프로젝트 생성 (이름, 도메인 선택)
3. 세션 생성 (mode: collaborative)
4. 목적 입력: "휴대폰 거치대를 자연물 컨셉으로 디자인"
5. 브리프 구조화 결과 확인

**예상 결과**:
- 프로젝트와 세션이 정상 생성
- 브리프에 purpose, domain, target, context, constraints 필드 포함
- AI가 누락 필드에 대해 추가 질문 생성

#### TC002: 자동 진행 모드
**테스트 목적**: 자동 모드 전체 파이프라인 실행 검증

**수행 단계**:
1. 세션 생성 (mode: auto)
2. 목적과 제약 입력
3. 자동 진행 대기
4. 상태 변화 모니터링 (queued -> researching -> concepting -> ... -> review_ready)

**예상 결과**:
- 각 상태 전환이 정상적으로 발생
- 자동 결정마다 점수, 근거, 대안, 리스크 저장
- 불확실성이 높은 항목에 "검토 필요" 표시
- 최종 결과에 중간 산출물 모두 포함

### 2.2. 사용자 스케치 기능

#### TC003: 스케치 업로드와 AI 해석
**테스트 목적**: 스케치 업로드, AI 분석, 사용자 승인 흐름 검증

**수행 단계**:
1. Sketch Input Board에서 스케치 파일 업로드
2. 스케치 유형 선택: rough
3. 설명 입력: "삼각형 실루엣 거치대 러프 스케치"
4. AI 분석 결과 확인
5. 해석 승인 또는 수정

**예상 결과**:
- 원본 파일이 불변으로 저장됨
- AI 분석에 intent, form_elements, structure, unclear_points 포함
- AI 해석이 "가설"로 표시됨
- 승인 후 다음 단계로 진행

#### TC004: 원본 보존 검증
**테스트 목적**: 스케치 원본이 구체화 결과로 덮어쓰이지 않음을 확인

**수행 단계**:
1. 스케치 업로드
2. 구체화 이미지 생성
3. 원본 파일과 구체화 결과 비교

**예상 결과**:
- 원본 파일 경로와 구체화 결과 경로가 다름
- 구체화 결과에 어떤 요소를 유지/변형했는지 설명 포함

### 2.3. 트렌드 조사 기능

#### TC005: 트렌드 근거 수집
**테스트 목적**: 출처 기반 트렌드 인사이트 수집 검증

**수행 단계**:
1. 브리프 완성 상태에서 트렌드 조사 시작
2. Evidence Board에서 인사이트 확인

**예상 결과**:
- 각 인사이트에 출처 URL, 발행일, 수집일 포함
- 출처가 불확실한 내용은 "가설" 또는 "검증 필요" 표시
- 같은 주장을 여러 출처가 반복하면 근거 강도 표시

### 2.4. 컨셉 관리 기능

#### TC006: 컨셉 후보 생성과 결정
**테스트 목적**: 컨셉 후보 생성, 평가, 결정 기록 검증

**수행 단계**:
1. 트렌드 조사 완료 후 컨셉 후보 생성
2. 각 후보 카드의 점수, 근거, 리스크 확인
3. 하나의 컨셉을 "채택" 선택
4. 다른 컨셉을 "폐기" 선택 (사유 입력)
5. Decision Log 확인

**예상 결과**:
- 각 후보에 점수와 근거 출처, 리스크 표시
- 채택/폐기/보류 결정이 Decision Log에 기록
- 폐기한 컨셉도 스펙 문서에 포함됨

### 2.5. 레퍼런스 기능

#### TC007: 레퍼런스 검색과 라이선스 관리
**테스트 목적**: 레퍼런스 검색, 출처 관리, 저작권 위험 표시 검증

**수행 단계**:
1. 컨셉 결정 후 레퍼런스 검색
2. Reference Board에서 결과 확인
3. 레퍼런스 카드의 출처, 라이선스, 위험 표시 확인
4. 레퍼런스 저장
5. 제외하는 경우 사유 기록

**예상 결과**:
- 각 레퍼런스에 출처 URL, 라이선스, 저작권 위험 표시
- 레퍼런스와 생성 이미지가 시각적으로 구분됨
- 제외 사유가 Decision Log에 기록됨

#### TC008: 레퍼런스 복제 금지
**테스트 목적**: 레퍼런스가 추상화를 통해서만 사용됨을 검증

**수행 단계**:
1. 레퍼런스 선택
2. 추상화 규칙 생성
3. 생성된 이미지가 원본 레퍼런스의 구도를 그대로 복제하지 않는지 확인

**예상 결과**:
- 추상화 규칙이 형태/구조/재질/상징 축으로 생성됨
- 생성 이미지는 원본 구도와 다름
- 원본 레퍼런스와의 연결이 추적 가능함

### 2.6. 추상화 기능

#### TC009: 추상화 규칙 생성
**테스트 목적**: 레퍼런스를 디자인 문법으로 변환 검증

**수행 단계**:
1. 선택된 레퍼런스로 추상화 요청
2. Abstraction Board에서 규칙 확인
3. 형태, 구조, 표면, 의미 축의 관찰-적용 매핑 확인

**예상 결과**:
- 최소 2개 이상의 축에서 규칙 도출
- 각 규칙에 원본 레퍼런스 출처 연결
- 사용자 스케치가 있는 경우 스케치 분석도 반영

### 2.7. 이미지 생성 기능

#### TC010: 추상화 기반 이미지 생성
**테스트 목적**: 추상화 규칙 기반 스케치/변형 생성 검증

**수행 단계**:
1. 추상화 규칙 선택
2. 이미지 생성 요청
3. 생성 결과 확인

**예상 결과**:
- 생성 이미지가 어떤 추상화 규칙에서 나왔는지 추적 가능
- 어떤 컨셉, 레퍼런스, 브리프와 연결되었는지 기록
- 사용자 스케치 기반 구체화 결과는 원본 스케치와 연결됨

#### TC011: 이미지 생성 전제 조건 검증
**테스트 목적**: 브리프/컨셉/추상화 없이 이미지 생성 차단 검증

**수행 단계**:
1. 브리프나 컨셉 없이 이미지 생성 시도

**예상 결과**:
- 에러 메시지: 최소 1개의 브리프, 컨셉, 추상화 규칙 필요

### 2.8. 스펙 문서 기능

#### TC012: 스펙 문서 생성
**테스트 목적**: 전체 파이프라인 결과의 스펙 문서화 검증

**수행 단계**:
1. 파이프라인 완료 후 스펙 문서 생성
2. 문서 섹션 확인

**예상 결과**:
- 프로젝트 브리프, 트렌드 근거, 컨셉 후보/결정, 레퍼런스 보드, 추상화 규칙, 생성 이미지, 최종안, 출처/라이선스 포함
- 버린 대안과 선택 사유 포함
- 사용자 스케치 원본과 AI 해석 포함

#### TC013: 스펙 문서 버전 관리
**테스트 목적**: 스펙 문서 버전 관리와 승인 워크플로우 검증

**수행 단계**:
1. 스펙 문서 v1 생성
2. 특정 단계 재실행
3. 스펙 문서 v2 생성
4. v1과 v2 비교

**예상 결과**:
- 두 버전이 모두 보존됨
- 변경 내용이 추적 가능함

### 2.9. 관리자 기능

#### TC014: 트렌드 출처 관리
**테스트 목적**: 관리자 콘솔의 트렌드 지식 관리 검증

**수행 단계**:
1. 관리자 콘솔 접속
2. 트렌드 출처 등록 (Core77, Dezeen)
3. 수집 주기 설정
4. 문서 수집 실행
5. 인사이트 생성 확인

**예상 결과**:
- 출처가 정상 등록됨
- 문서 수집 후 파싱 결과 저장
- 인사이트에 출처와 발행일 연결

#### TC015: 모델 카탈로그 관리
**테스트 목적**: 기능별 AI 모델 정책 관리 검증

**수행 단계**:
1. Model Provider 등록
2. Model Catalog에 모델 추가
3. Feature Model Policy 설정 (기능별 모델 지정)
4. Fallback 정책 설정

**예상 결과**:
- 모델 정책이 정상 저장됨
- Fallback 시 거짓 결과가 아닌 실패 보고 + 대체 모델 사용

## 3. SaaS 멀티테넌시 테스트

#### TC016: 데이터 격리
**테스트 목적**: 테넌트 간 데이터 격리 검증

**수행 단계**:
1. 테넌트 A에서 프로젝트 생성
2. 테넌트 B에서 테넌트 A의 프로젝트 접근 시도

**예상 결과**:
- 테넌트 B에서 테넌트 A의 데이터 접근 불가
- AuditLog에 접근 시도 기록

## 4. 파이프라인 불변 조건 테스트

#### TC017: 출처 없는 트렌드 근거 사용 금지
**수행 단계**:
1. 출처 없는 트렌드 인사이트가 컨셉 결정 근거로 사용되는지 확인

**예상 결과**:
- 출처 없는 인사이트는 "가설"로 표시
- 컨셉 결정 근거에 "가설"이 포함되지 않음

#### TC018: 레퍼런스와 생성 이미지 구분
**수행 단계**:
1. 레퍼런스와 AI 생성 이미지를 나열
2. UI 라벨, 저장 위치, 카드 색상 확인

**예상 결과**:
- 두 타입이 시각적으로 명확히 구분됨

## 5. 사용성 테스트

#### TC019: 현재 단계 파악
**테스트 목적**: 사용자가 항상 현재 단계, 다음 결정, 근거를 파악할 수 있는지 확인

**수행 단계**:
1. 파이프라인 중간 단계에서 화면 확인
2. Decision Panel의 현재 단계, 다음 결정 표시 확인

**예상 결과**:
- 상단에 현재 단계 표시
- Decision Panel에 다음 결정과 이유 표시
- AI 제안 옆에 근거 출처 링크 표시

## 6. 합격 기준

- **기능 테스트**: 100% 합격
- **불변 조건 테스트**: 100% 합격
- **멀티테넌시 격리**: 100% 합격
- **사용성**: 사용자가 현재 단계, 다음 결정, 근거를 항상 파악 가능
