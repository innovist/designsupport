# Plan 21: 실패 세션 옵션 기반 최소 크롤링 실제 파이프라인 테스트

**작성일**: 2025-12-23 03:40  
**목적**: 기존 실패 세션의 옵션을 바탕으로 최소 크롤링으로 전체 파이프라인(백엔드/프론트) 실제 실행을 검증하고, 발생 오류를 즉시 수정 후 재검증한다.

---

## 0. 전제/제약
- Git 저장소 부재로 브랜치/PR 단계 수행 불가.
- 파일 ≤ 300 LOC, 함수 ≤ 50 LOC, 매개변수 ≤ 5, 순환 복잡도 ≤ 10 준수.
- 변경 전 관련 파일 전체 읽기(부분 읽기 금지).
- 테스트는 실제 기능 실행 필수, 로그/결과 증빙.

## 1. 핵심 구성 요소 및 기대 기능
- 세션 생성 옵션: 기존 실패 세션(필터/크롤러/기간) 기반.
- 최소 크롤링: max_items_per_source 등 최소화.
- 파이프라인: 입력→키워드→크롤링→분석→이미지(+블루프린트 스킵 가능)까지 실제 호출.
- 프론트: /projects 페이지 로드 및 개요 로그 영역 렌더링 확인.

## 2. 엣지 케이스
- 외부 API(키워드/분석/이미지) 실패 또는 쿼터 제한.
- YouTube STT 전사 실패(모델/네트워크/다운로드 오류).
- 크롤링 결과 0건.
- 서버 미기동 또는 포트 충돌.

## 3. 데이터 흐름/영향도
- /api/v1/sessions POST → session_store.start_pipeline → orchestrator.run_complete_pipeline.
- progress_cb/state_cb → sessions.metadata_json logs/메타 업데이트 → /api/v1/sessions/{id}/logs.
- /projects 템플릿 + overview.js → 개요 탭 터미널 로그 렌더링.

## 4. 대안 비교(결정 기록)
1) TestClient로 API 호출
   - 장점: 서버 실행 불필요
   - 단점: 백그라운드 task 지속성/프론트 확인 어려움
   - 위험: 파이프라인 실행 불완전
2) 실제 서버 실행 후 HTTP 테스트
   - 장점: 백엔드/프론트 통합 검증 가능
   - 단점: 실행 시간이 길어짐
   - 위험: 외부 API 실패 가능

→ 선택: 2) 실제 서버 실행 + HTTP 테스트

## 5. 구현/테스트 단계
1) 기존 실패 세션 옵션 확인 및 누락 옵션 결정(최소 수집값).
2) 서버 기동 → /health, /projects 등 프론트 확인.
3) 세션 생성(기존 옵션 기반, 최소 크롤링) → 파이프라인 자동 실행.
4) 상태/로그/결과 폴링, 실제 수집/분석/이미지 생성 확인.
5) 오류 발생 시 원인 추적→수정→정적 검증→재실행.

## 6. 완료 기준
- 세션이 completed 또는 명확한 오류 원인과 수정 완료.
- 크롤링 로그가 개요 탭 API 로그로 확인됨.
- 실제 수집/분석/이미지 결과 또는 실패 시 근거 로그 확보.
